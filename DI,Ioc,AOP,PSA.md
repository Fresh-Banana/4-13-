## Spring Framework(스프링 프레임 워크)
### `정의 :` 자바 기반으로 만든 웹 애플리케익션 프레임 워크
### `프레임 워크:` 개발을 용이하게 만드는 기본 뼈대
* 스프링에서는 Framework(뼈대)를 제공해주면서 개발을 용이하게 함

## spring Contatiner(스프링 컨테이너)
* `Bean`-> 스프링이 만든 객체
* `Container`->스프링이 객체를 관리하는곳
* `DI` -> 스프링이 객체를 직접 넣어주는것 

## 프레임 워크의 모델
* IoC/DI(개발 규약)
    * IoC(제어의 역전),의존성(DI)
* 서비스 추상화
* AOP(독립적인 모듈화,겹치는 로직(흐름,절차)을 하나로)
* API
## AOP(관점 지향적 프로그래밍) -> 겹치는부분을 묶음 
`정의 :` 횡단 관심사를 자동으로 처리해줌

### 종단(세로) 관심사
* Controller -> Service -> Repository
    * ex) "주문하기", "결제하기", "회원가입하기등은 **각 클래스에 딱 맞는 일이있음**
### 횡단 관심사
* 로깅,트랜잭션등은 Controller -> Service -> Repository 이 모든것에 **적용되는 기술임**(세로로 흐르는 화살표로 가로질러가서 이렇다)
    * 로그 남기기", "트랜잭션 처리", "권한 체크"같은것은 **언제 어디서나** 필요함
    * 이걸 모두 복붙해서 프로그램 앞에 놓으면 머리가 펑펑임
## IoC/DI(개발 규약)
* ### 제어권 역전을 이용하여 뭐가 가능할까?
1. 객체 생성 밎 관리
    * 프레임 워크가 객체를 생성함(컨테이너에서)
    * 이를 토대로 프레임워크가 **생명주기**를 제어함
2. 의존성 주입
    * 객체를 생성할때 필요한 객체를 직접 생성함
    * 객체 생성의 주도권을 줬기때문에 가능함
    * 더욱더 편하게 객체를 생성할수있게 됌
3. 애플리케이션의 생명주기 관리 
    * 어노테이션을 적용하여 컨테이너가 각 코드의 생명주기를 관리함
    * 제어권을 주었기때문에 메모리,삭제 등을 생각 X -> 객체가(로직) 하는 일에만 집중할수있음
4.  웹 요청 처리
    *  개발자가 웹 처리 방식을 직접 제어하않고 스프링이 요청을 컨트롤러에 전달하고 이후 흐름을 관리해줌
* 결합도가 감소하고 유연성을 높이고
  
### DI
* 의존객체를 다른 객체(외부)에서 넣어주는것
### DIP
* DIP는 메인 모듈이 바로 하위모듈에 의존성을 주입하는것이 아닌 중간 모듈인 interface라는 모듈을 구현하여 의존성을 주입하는 방식이다
* 메인 -> interface
* interface <- 하위
## PSA
* 일관된 복잡한 코드를 하나로 묶는 개념
* ex) 트랜잭션의 처리 방식을 @@Transactional로 묶는
* ex) 인터페이스를 사용(기능과 구현을 따로) 

## 계층형 아키텍처(Layered Architecture)
* Presentation,Service,data Access 계층으로 나뉨
    * 이 계층들을 각자 계층에 맞는 코드를 작성함
* presentation - 응답을 주고,데이터를 돌려주는 계층
* Service - 비지니스 로직(내가 생각하느 프로그램의 동작)을 작성하는 계층
* Data Access - 데이터를 저장,조회등 **DB**에 접근하는 계층

## 계층별로 하는 일을 나누면 관심사로 나눌수 있음
* 겹치는것(횡단(세로)),안겹치는것(종단(가로))

## 서블릿의 동작
1. 클라이언트의 요청으로 부터 시작함(URL)
    * 이 요청을 HTTP REQUEST(요청이라함)
    * 이 요청을 받을때에는 한가지의 상자로만 받음
    * 즉 무상태성임
2. URL에 있는 요청(/hell o)을 확인함
    * hello라는 이 클래스가 있나 찾아봄
    * 톰캣이 미리 지도를 그려줌
3. 그 지도로 hello라는 요청에 맞는 서블릿(@WebServlet("/order"))을 찾음
4. service는 위에서 찾은것을 바탕으로 역할을 나눠줌
    * get - doGet
    * post - doPost
5. service가 나눠준 역할을 실행함
* `정리 :` 요청 -> 요청확인 -> 한번 탐색 -> 서블릿 발견 -> 요청 분배 -> 알맞는 작업 호출 및
