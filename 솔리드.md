## SRP(single Resposibility principle)
* `정의 :` 클래스를 통하여 수행하는 대표적인 업무는 하나여야 한다는 뜻
    * 하나의 책임을 위해 여러 메서드가 사용될수있음
```java
   public class UserService {

    public void saveUser(User user) {
        // Save user to database
        System.out.println("User saved to database: " + user.getName());
    }

    public void sendWelcomeEmail(User user) {
        // Send welcome email to user
        System.out.println("Welcome email sent to: " + user.getEmail());
    }

    public void logUserActivity(User user) {
        // Log user activity
        System.out.println("Logging activity for user: " + user.getName());
    }
}
```
* 위 클래스처럼 하나의 클래스 안에 다양한 책임이 있으면 코드를 유지보수도 힘들 뿐더러 코드를 변경할때에는 의도치 않은 영향을 다른 메서드에 미칠수도 있다.
* 클래스 안에서 기능을 빼서 사용하기도 어렵다
* 그래서 위 코드의 기능을 하나 하나 클래스로 만들어야한다.

## OCP(Open Closed Principle)
* 개방/폐쇄 원칙
* 각 클래스가 **확장**에는 **열려있고** **변경**에는 **닫혀 있어야한다**라는 원칙
  
```java
public class ReportGenerator {

    public void generateReport(String type) {

        if (type.equals("PDF")) {
            System.out.println("Generating PDF report...");
        } else if (type.equals("HTML")) {
            System.out.println("Generating HTML report...");
        }

    
    }
}
```
* 위코드에서 추가로 다른 형식을 추가하면 의도치 않은 영향과 코드가 장황해짐

```java
public interface Report {
    void generate();
}

public class PDFReport implements Report {
    @Override
    public void generate() {
        System.out.println("Generating PDF report...");
    }
}

public class HTMLReport implements Report {
    @Override
    public void generate() {
        System.out.println("Generating HTML report...");
    }
}

public class XMLReport implements Report {
    @Override
    public void generate() {
        System.out.println("Generating XML report...");
    }
}
```
* 그래서 이처럼 interface를 만든후 메서드를 오버라이딩하여 표현한다
* 확장성이 용의해지고 코드 수정도 쉬워지는 장점이 있다

## LSP(Liskov Substitution Principle)
* 자식클래스는 언제나 부모 클래스를 대체할수 있어야한다.
```java
class Square extends Rectangle {
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width;
    }

    @Override
    public void setHeight(int height) {
        this.width = height;
        this.height = height;
    }
}
class AreaCalculator {
    public void calculateArea(Rectangle rectangle) {
        rectangle.setWidth(5);
        rectangle.setHeight(4);
        System.out.println("Area: " + rectangle.getArea());
    }
}
```

* 위 클래스에서 직사각형은 정사각형의 일종이라 괜찮다 생각할수 있지만 저렇게 되면 직사각형의 변의 길이를 조정하는 코드는 없다

## ISP(Interface Segregation)
* 클래스는 자신이 사용하지 않을 메서드를 강요받지 않아야 한다.
```java
// Interface for workers
public interface Worker {
    void work();
    void eat();
}

// Class representing a regular worker
public class Employee implements Worker {
    @Override
    public void work() {
        System.out.println("Employee is working");
    }

    @Override
    public void eat() {
        System.out.println("Employee is eating");
    }
}
// 사람은 먹기도하고 일도 하지만 

public class Robot implements Worker {
    @Override
    public void work() {
        System.out.println("Robot is working");
    }

    @Override
    public void eat() {
        // Robots do not eat
        throw new UnsupportedOperationException("Robots do not eat");
    }
}


public class Main {
    public static void main(String[] args) {
        Worker employee = new Employee();
        employee.work(); // Employee is working
        employee.eat();  // Employee is eating

        Worker robot = new Robot();
        robot.work(); // Robot is working
        robot.eat();  // Throws UnsupportedOperationException
    }
}
// 로봇은 먹지를 않는데 구현하게 되어있다.
```
* 그래서 eat 메서드와 work메서드를 따로 만들어야한다
```java
public interface Worker {
    void work();

}
public interface eat {
    void eat();

}
``` 
## DIP(Dependenecy Inversion Principle)
* 고수준 모듈이 저수준 모듈에게 의존해서는 안된다라는 뜻을 갖고있다.

## extend랑 implement와 합성의 차이점 
* `extend :`부모의 클래스 **재사용**에 중점을 둠, 또한 부모의 기능을 다 갖고있고 기능을 추가할수도 있음.
* `implement :`**@overriding**으로 메서드를 구현해야할 의무를 부여함 이런 규칙을 지켜라는 의미로 씀
```java
// Interface for switchable devices
public interface Switchable {
    void turnOn();
    void turnOff();
}
// Low-level class implementing the interface
public class Fan implements Switchable {

    @Override
    public void turnOn() {
        System.out.println("Fan is spinning");
    }

    @Override
    public void turnOff() {
        System.out.println("Fan is stopping");
    }
}
// High-level class
public class Switch {
    private Switchable device;

    public Switch(Switchable device) {
        this.device = device;
    }

    public void turnOn() {
        device.turnOn();
    }

    public void turnOff() {
        device.turnOff();
    }
}
```
* 고수준이 저수준 모듈에 의존 받지않아야함 
* 그래서 고수준 모듈은 따로 구현하고 저수준 모듈도 따로 구현함
